Please create a comprehensive technical specification document at `docs/technical-spec.md` that will serve as the single source of truth for all developers working on this project. The document should enable developers to work independently on different components while ensuring seamless integration without conflicts.

The technical specification must include the following sections:

1. **Project Overview**
   - Brief summary of the project purpose and scope
   - Technology stack and dependencies
   - Development environment setup requirements

2. **Project Structure**
   - Complete directory and file structure
   - Purpose and responsibility of each major file/directory
   - Module organization and separation of concerns

3. **API Specifications**
   For each API endpoint, provide:
   - Endpoint URL and HTTP method
   - Request format (headers, query parameters, request body with exact field names and types)
   - Response format (status codes, response body structure with exact field names and types)
   - Authentication/authorization requirements
   - Error responses and error codes
   - Example requests and responses

4. **Business Logic & Sequence Diagrams**
   - Mermaid sequence diagram for each API endpoint showing the complete flow from request to response
   - Step-by-step business logic description for each endpoint
   - Validation rules and business constraints
   - Integration points between different components/services

5. **Data Schema**
   - Complete database schema (if applicable) with table definitions, field types, constraints, and relationships
   - Data models/entities used throughout the application with all properties and their types
   - Mermaid entity-relationship diagram (ERD) showing relationships between data entities
   - Data validation rules and constraints

6. **Architecture Diagram**
   - High-level Mermaid diagram showing system architecture and component interactions
   - Data flow between frontend, backend, database, and external services

7. **Integration Contracts**
   - Interface definitions for component-to-component communication
   - Shared types, constants, and enums
   - Event schemas (if using event-driven architecture)

8. **Development Guidelines**
   - Coding standards and conventions
   - Naming conventions for files, functions, variables
   - Git workflow and branching strategy

Please analyze the existing codebase thoroughly to extract all relevant information and ensure the specification accurately reflects the current implementation and design patterns being used.
